# [team-devs-ai] Pipeline de truncamento de contexto e FinOps

**Fase:** 4 — Configuração  
**Labels:** config, cost, memory

**Implementação:** ConfigMap [k8s/security/finops-config-configmap.yaml](../../k8s/security/finops-config-configmap.yaml); scripts [truncate_payload_border.py](../../scripts/truncate_payload_border.py), [working_buffer_ttl.py](../../scripts/working_buffer_ttl.py), [context_validation_hook.py](../../scripts/context_validation_hook.py), [compact_preserve_protected.py](../../scripts/compact_preserve_protected.py); exemplos em [docs/agents-devs/](../agents-devs/) (SESSION-STATE.example.md, microADR-template.json, CRITERIOS_ACEITE-example.md). Validação: [validacao-040-041-completa.md](validacao-040-041-completa.md).

## Descrição

Causa raiz do custo é inchaço de contexto. Implementar pipeline de truncamento e sumarização com **controle determinístico na infraestrutura** (Gateway e Redis), não delegar ao LLM nem depender de agente para "limpar" contexto: **max tokens por request no Gateway** (perfil CEO e demais que usam nuvem); **pre-flight Summarize** obrigatório para issues/conversas com &gt;3 interações (modelo local, antes do envio à nuvem); **truncamento na borda** (script na entrada do stream); **TTL no Redis** para expirar mensagens antigas do working buffer (sem depender do DevOps); janela deslizante e memória duas camadas + RAG; freio de emergência (ex.: $5/dia). Configurar limite rígido de gastos no painel do provedor.

## Critérios de aceite

- [ ] Documentação da causa raiz (contexto não filtrado enviado aos LLMs em nuvem; risco de OOM se LLM tentar resumir logs gigantes). Quando o contexto ou RAG para o PO está errado ou desatualizado, há risco de **alucinação de escopo** (tarefas impossíveis na base atual); ver [02-agentes.md](../02-agentes.md).
- [ ] **Max tokens per request no Gateway:** limite rígido configurado no Gateway (config JSON/YAML) por perfil de agente (CEO e demais que usam nuvem), aplicado **antes** da requisição chegar ao provedor (Gemini, etc.).
- [ ] **Pre-flight Summarize obrigatório:** para issues/conversas com **mais de três interações** destinadas à nuvem, orquestrador intercepta o payload, modelo **local** gera resumo executivo denso, substitui histórico bruto no payload; só então o Gateway envia à nuvem. Evita corte no meio de JSON; reduz carga paga.
- [ ] **Truncamento na borda:** script (ex.: Python) **antes de enfileirar** no Redis Stream: conta tokens do payload (ex.: limite 4000 tokens); se exceder, truncamento bruto (cabeçalho do erro + causa raiz no final; remove miolo do log/stack trace). Corte seco **não** deve agir sozinho (risco de corromper JSON); aplicar após pre-flight quando aplicável. DevOps e demais agentes **só recebem** payload já limitado; máquina nunca recebe carga que estoure VRAM.
- [ ] **TTL no Redis para working buffer:** configurar TTL nas chaves do buffer de conversas para que mensagens antigas **expirem automaticamente**; "lixo digital" evapora **sem depender do agente DevOps** para limpar.
- [ ] Pipeline de truncamento/sumarização descrito ou implementado (janela deslizante; sumarização via pre-flight para &gt;3 interações; modelo local apenas sobre payload já truncado na borda).
- [ ] Uso de memória em camadas e RAG para reduzir tokens enviados (integrar com memória Elite).
- [ ] **MicroADR:** Architect gera **microADR** (JSON estrito, ex.: via markitdown/saída formatada) ao aprovar PR; microADR **nunca** é sumarizado e é anexado **diretamente** à memória vetorial de longo prazo (Warm Store). Ver [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md) (seção 2.3) e [28-memoria-longo-prazo-elite.md](../28-memoria-longo-prazo-elite.md).
- [ ] **Invariantes de negócio:** lista no arquivo principal de estado da sessão (ex.: SESSION-STATE); regras absolutas do Diretor com **tag especial**; script de limpeza do DevOps com **regex** para **nunca** apagar linhas com essa tag. Ver [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md) (seção 2.3).
- [ ] **Gancho de validação de contexto (operado localmente, antes de mandar para a sumarização na nuvem):** objetivo focado na **preservação do valor de negócio** (evitar amnésia funcional). Inserir etapa no pipeline em que um **modelo local e gratuito** (ex.: Llama 3, já usado no Architect) faz **varredura no buffer de trabalho** buscando **exclusivamente intenções do usuário ou regras informais que não ganharam tag**. Se o modelo local achar algo crítico, **propor extração direto para o arquivo principal de estado (Session State)** — protege o escopo sem gastar API. Ver [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md) e [28-memoria-longo-prazo-elite.md](../28-memoria-longo-prazo-elite.md).
- [ ] **Segregação dos critérios de aceite (obrigatória para validação reversa):** Os critérios de aceite originais **não** podem estar no mesmo buffer que é sumarizado — senão o PO recebe só o resumo e perde a referência para comparar. Implementar **uma** das opções: **(A) Tag de proteção:** critérios de toda issue com tag especial no Markdown (ex.: `<!-- CRITERIOS_ACEITE -->`); script de limpeza/compacted do DevOps configurado com **regex** para **ignorar** qualquer bloco dentro dessa tag (nunca apagar nem enviar ao sumarizador). **(B) Payload duplo:** orquestrador armazena critérios em **arquivo separado** (ex.: SESSION-STATE.md ou session.md por issue) e envia à nuvem **resumo + critérios intactos**; PO sempre recebe os dois. Ver [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md) (seção 2.2), [28-memoria-longo-prazo-elite.md](../28-memoria-longo-prazo-elite.md) e [12-ferramenta-summarize.md](../12-ferramenta-summarize.md).
- [ ] **Validação reversa (PO):** logo **após a sumarização**, o PO executa validação reversa: **pega o resumo recém-gerado e compara contra os critérios de aceite originais** das tarefas (sempre disponíveis intactos via segregação acima). **Se o resumo omitir um critério fundamental, o PO rejeita o truncamento.** O sistema é forçado a **reestruturar o bloco de dados** (ex.: manter trechos não sumarizados ou refazer o resumo). Ver [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md) e [02-agentes.md](../02-agentes.md) (PO).
- [ ] Freio de emergência: interromper e alertar se custo de API exceder threshold (ex.: $5/dia). Controle primário sustentável: **token bucket** para eventos de estratégia e **degradação por eficiência** (razão CEO/PO); ver [126-token-bucket-degradacao-eficiencia.md](126-token-bucket-degradacao-eficiencia.md) e [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md).
- [ ] **Acelerador preditivo:** quando o sistema prever estouro de orçamento (ex.: por tamanho do diff ou histórico de tokens), rotear tarefa para **modelo local em CPU** em vez de acionar apenas o freio; documentar em [05-seguranca-e-etica.md](../05-seguranca-e-etica.md), [07-configuracao-e-prompts.md](../07-configuracao-e-prompts.md) e [22-modelos-gratuitos-openrouter-freeride.md](../22-modelos-gratuitos-openrouter-freeride.md).
- [ ] **Persistência acoplada ao FinOps:** contador de tentativas de resolução integrado ao Gateway FinOps; no pre-flight, custo estimado × número da tentativa (penalidade progressiva); ao atingir gatilho, orquestrador interrompe a tarefa, devolve ao backlog do PO e libera travas de hardware (GPU). Ver [06-operacoes.md](../06-operacoes.md), [13-habilidades-proativas.md](../13-habilidades-proativas.md).
- [ ] Recomendação de limite de gastos no painel do provedor (Google Cloud, etc.) documentada.
