## Project Overview

**ClawDevs AI** is a full-stack AI agent orchestration platform. It runs a local stack of 10 Docker containers: a Next.js control panel, a FastAPI backend, an OpenClaw agent gateway, Ollama for local LLM inference, PostgreSQL (with pgvector for RAG), Redis (queue + cache), SearXNG (web search), and supporting services.

The Makefile is the primary way to operate the stack. There is no docker-compose.yml.

## Essential Commands

### Stack Operations
```bash
make up-all           # Build + start all 10 containers (first run)
make up-all-with-cache # Start all with Docker layer cache
make down             # Stop all containers
make restart          # Stop + restart
make status           # Container status
make logs             # Aggregated logs
make logs-follow      # Real-time logs
make env-check        # Validate .env before starting
```

### Individual Services
```bash
make up-<service>     # e.g., make up-postgres, make up-openclaw
make openclaw-restart # Restart OpenClaw agent gateway (no rebuild)
```

### Database
```bash
make panel-db-migrate # Run Alembic migrations
```

### Cleanup
```bash
make reset            # Remove containers + volumes
make destroy          # Remove containers, images, cache
```

### Frontend Development
```bash
cd control-panel/frontend
pnpm install
pnpm dev              # Next.js with Turbopack at localhost:3000
pnpm build            # Production build
pnpm lint             # TypeScript type check (tsc --noEmit)
pnpm orval            # Regenerate API clients from OpenAPI spec
```

### Backend Development
```bash
cd control-panel/backend
uv sync               # Install Python deps
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### Tests
```bash
cd control-panel/backend
uv run pytest tests/            # All tests
uv run pytest tests/test_foo.py # Single test file
```

## Architecture

### Request Flow
1. Browser ŌåÆ Next.js frontend (`:3000`)
2. Next.js rewrites `/api/*` ŌåÆ FastAPI backend (`:8000`)
3. React Query clients (auto-generated by Orval from `/openapi.json`) call backend
4. Long-running work ŌåÆ Redis ŌåÆ RQ workers
5. Real-time updates pushed via WebSocket (`/ws`)

### Backend Structure (`control-panel/backend/app/`)
- `main.py` ŌĆö FastAPI app with lifespan (runs migrations, bootstraps admin, syncs agents, starts health monitor)
- `routers/` ŌĆö Route handlers (auth, agents, sessions, tasks, approvals, memory, crons, metrics, cluster, etc.)
- `models/` ŌĆö SQLModel ORM models (maps to PostgreSQL tables)
- `services/` ŌĆö Business logic (agent sync, RAG, OpenClaw client, governance, circuit breaker, cost tracker)
- `tasks/` ŌĆö RQ async tasks (periodic sync, task orchestration)
- `core/` ŌĆö Config (pydantic-settings), database session factory, auth utilities
- `hooks/` ŌĆö OpenClaw tool execution callbacks

### Frontend Structure (`control-panel/frontend/`)
- App Router (Next.js 16) with Turbopack
- `src/lib/api/` ŌĆö Orval-generated React Query + Axios clients
- `src/lib/ws.ts` ŌĆö WebSocket client
- `next.config.ts` ŌĆö Rewrites `/api/*` and `/api/ws/*` to the backend

### Data Layer
- **PostgreSQL 15** with **pgvector** ŌĆö primary store + vector embeddings for RAG
- **Redis** ŌĆö RQ task queue and caching
- **Alembic** ŌĆö migrations (auto-run at backend startup)

### Agent Communication
- **OpenClaw Gateway** (`:18789`) ŌĆö receives agent tool calls, executes them, returns results
- `services/openclaw_client.py` ŌĆö backend HTTP client to the gateway, authenticated with `OPENCLAW_GATEWAY_TOKEN`
- Agents run on configurable cron schedules (env vars like `DEV_BACKEND_CRON_EXPR`)

### LLM Providers
- **Ollama** (local, `:11434`) ŌĆö default
- **OpenRouter** (cloud) ŌĆö switch via `PROVEDOR_LLM=openrouter` in `.env`

## Environment Setup

Copy `.env.example` to `.env` and configure:
- `OPENCLAW_GATEWAY_TOKEN` ŌĆö shared internal auth token
- `PANEL_SECRET_KEY` ŌĆö JWT secret (run `make gen-secret`)
- `PANEL_ADMIN_USERNAME` / `PANEL_ADMIN_PASSWORD` ŌĆö control panel login
- `PANEL_DB_PASSWORD` / `PANEL_REDIS_PASSWORD` ŌĆö service passwords
- `GIT_TOKEN` / `GIT_ORG` ŌĆö GitHub integration
- `TELEGRAM_BOT_TOKEN_CEO` / `TELEGRAM_CHAT_ID_CEO` ŌĆö notifications
- `PROVEDOR_LLM` ŌĆö `ollama` or `openrouter`

## Key Conventions

- **Frontend package manager**: pnpm ŌĆö always use `pnpm`, never `npm` or `yarn`
- **Backend package manager**: uv ŌĆö always use `uv run` / `uv sync`, never `pip`
- **API client generation**: After changing backend schemas, run `pnpm orval` in the frontend to regenerate TypeScript clients
- **Migrations**: Model changes require a new Alembic migration; they run automatically on backend startup
- **Documentation language**: README and docs are in Portuguese (pt-BR); code is in English

## Project

**ClawDevs AI Panel UI Modernization (Mosaic)**

This project modernizes the existing `control-panel/frontend` experience in a brownfield codebase. The goal is to adopt the visual/system patterns from Cruip's Tailwind dashboard template (Mosaic) while preserving current application capabilities, routes, and backend integrations.

The current panel already runs on Next.js App Router, Tailwind CSS v4, and chart components; this initiative upgrades visual consistency and dashboard UX without changing the product domain.

**Core Value:** Operators can monitor and manage AI workflows quickly from a consistent, fast, and reliable dashboard interface without losing existing functionality.

### Constraints

- **Tech Stack**: Keep Next.js + React + Tailwind 4.x and existing frontend architecture
- **Compatibility**: Preserve current route map and backend API contracts
- **Quality**: Avoid regressions in existing task/chat/session/monitoring flows
- **Security**: No secret leakage from `.env` or runtime configs during migration
- **Incremental Delivery**: Execute in phased steps to keep deployability throughout

## Conventions

## Frontend Conventions
- TypeScript-first React codebase in `control-panel/frontend/src`.
- App Router pages in `src/app/**/page.tsx`.
- Shared UI atoms in `src/components/ui/*`.
- Domain grouping pattern in `src/components/{dashboard,monitoring,approvals,...}`.
- Utility helpers centralized in `src/lib`.
- Styling stack assumes Tailwind utility classes plus helper libs (`clsx`, `tailwind-merge`).
## Backend Conventions
- Router modules segmented by capability in `control-panel/backend/app/api/*.py`.
- Test module naming follows `test_*.py` under `control-panel/backend/tests`.
- Async endpoints and service logic aligned with FastAPI + AnyIO stack.
- Migrations managed in `control-panel/backend/migrations`.
## Naming and Organization Patterns
- Feature-based naming for page routes (`chat`, `tasks`, `sessions`, `monitoring`).
- UI component names are descriptive and usually noun-based (`stats-card.tsx`, `usage-chart.tsx`).
- API files map 1:1 to domain concepts (`approvals.py`, `repositories.py`, `settings.py`).
## Tooling and Quality Hints
- Frontend scripts emphasize type-safety (`npm/pnpm run lint` wired to `tsc --noEmit`).
- Backend typing checks indicated by mypy cache and config in `pyproject.toml`.
- Cypress and pytest are both present and already used.
## Migration-Specific Convention Guidance
- Reuse existing route/component boundaries instead of collapsing everything into one page.
- Keep chart data plumbing consistent with existing abstractions in `src/components/dashboard` and `src/components/monitoring`.
- Preserve `src/components/layout/*` composition contracts while replacing visuals.
